# i18n

Every user-facing string is translated into **all five locales**. This is a hard
rule, not a nicety: a missing key is a gate-level defect.

## The five locales

`@nuxtjs/i18n` 10.4.0 with files in `i18n/locales/`:

| Code | Language | Dir | Notes |
|---|---|---|---|
| `en` | English | ltr | the base; write keys here first |
| `fr` | French | ltr | |
| `th` | Thai | ltr | drives the Noto Sans Thai font bundling in share images |
| `tlh` | Klingon | ltr | terse and in-character; it is itself an easter egg, see [../features/easter-eggs.md](../features/easter-eggs.md) |
| `ar` | Arabic | **rtl** | the first right-to-left locale; see [rtl.md](rtl.md) |

Each locale entry in `nuxt.config.ts` carries `code`, `name`, `file`, a `language`
(BCP47 tag, used for `<html lang>`) and, for Arabic, `dir: 'rtl'`. The active
locale is auto-detected and persisted in the `ng_locale` cookie.

## Right-to-left

Arabic is rendered right-to-left. The mechanics (dynamic `<html dir>`, the
physical-to-logical CSS convention, the mirrored bracket, icon flipping) live in
their own doc: **[rtl.md](rtl.md)**. The short version: write direction-agnostic
CSS (logical utilities `ms/me/ps/pe`, `text-start/end`, `border-inline-*`), never
physical `ml/mr/text-left/...`, so a new component mirrors for free.

## Key organization

Keys are nested by feature module (`chat.*`, `verify.*`, `leagues.*`,
`update.*`, `push.*`, etc.) with subkeys for titles, descriptions, button labels,
errors and interpolated strings (`{placeholder}`). When you add a feature, add
its block to every locale file in the same change.

## The changelog

The `/about` changelog also renders in the active locale, but its copy is not in
the JSON locale files. `CHANGELOG.md` is the canonical English source (Keep a
Changelog, also the release tooling's input); each translation is a parallel
Keep-a-Changelog file in `i18n/changelogs/{fr,th,tlh,ar}.md`, mirroring it
bullet-for-bullet (version headers and dates identical, section titles and
bullets translated). `useChangelog` imports all five with `?raw`, parses each,
and overlays the active locale onto English per version, falling back to the
English entry for any version a locale hasn't translated yet (so nothing ever
goes blank). English owns the version list, ordering, and the "unseen" badge -
all locale-independent. The same all-locales rule applies: a new `CHANGELOG.md`
entry must be added to every locale file. `mise run changelog check` enforces
structural parity (same versions, sections, bullet counts) and runs in the gate
and the release; `mise run changelog promote` moves `[Unreleased]` in every file
together. See [../features/changelog.md](../features/changelog.md).

## Server-side copy

Most strings are resolved client-side, but some text is produced on the server
where the client `itemText` helpers cannot run:

- **Push notification copy** lives in `server/utils/push/content.ts` (`push.*`
  keys) so a push can be localized at send time. See
  [../features/web-push.md](../features/web-push.md).
- Share-card and `/verify` strings are likewise resolvable server-side
  (`server/utils/share/i18n.ts`).

Both server dicts import the locale JSON directly (single source of truth), so
adding `ar` there is just an extra import + registry entry, not a second
translation. The share-locale set is `SHARE_LOCALES` in
`server/utils/share/token.ts` (drives `ShareLocale`, `FEED_LOCALES`, and the
OpenAPI `enum`s), and it now includes `ar`.

## The rule, restated

- New user-facing string -> add the key to `en`, `fr`, `th`, `tlh` AND `ar`.
- Keep `tlh` terse and in-character.
- Write direction-agnostic CSS so it mirrors under `ar` - see [rtl.md](rtl.md).
- Informational tooltips are i18n'd and use the PrimeVue `v-tooltip` directive,
  never a native `title=` (the one exception is a `title=` that only reveals text
  cut off by `truncate`). See [client.md](client.md).
